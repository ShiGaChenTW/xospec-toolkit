Why This Architecture

設計背景：AI Coding 的五個核心問題

這套架構不是憑空設計——它是針對 AI Agent 輔助開發時反覆出現的真實痛點，逐一修正而成。

#問題症狀引入什麼來修正
1 Context 遺失 Agent 不記得上輪對話，反覆問同樣的問題 X.OSpec Spec 活在 repo 裡，Agent 每次開工先讀檔案，不依賴 chat 記憶
2 Scope 漂移 要求做 A，Agent 自作主張加了 B、C、D X.OSpec proposal + tasks 嚴格定義「只做什麼」，AGENTS.md 規定先 flag scope drift
3 品質失控 沒測試、沒 review，直接宣稱完成 Superpowers TDD、code-review、verification skills 強制插入品質關卡
4 執行無序 Agent 隨意跳著做，無法追蹤進度 GSD ROADMAP + Phase 工作流提供結構化的執行順序與進度追蹤
5 跨工具不可攜 換 IDE 就要重新設定所有 workflow 三者整合 全部是純 Markdown 存在 repo 裡，跨 Cursor / Copilot / Kiro 通用

三框架各管什麼？

X.OSpec

管「做什麼」

specs/ 定義系統應然狀態，changes/ 記錄每次變更的完整包，spec delta 確保規格永遠最新

Superpowers

管「怎麼做得好」

brainstorming → writing-plans → TDD → code-review → verification，在 Agent 決策迴路插入品質關卡

GSD

管「按什麼順序做」

PROJECT.md 全域上下文、ROADMAP.md 拆 milestone → phase、STATE.md 追蹤進度與決策

關鍵設計決策：選擇「repo 檔案驅動」而非「IDE 一條龍」。真正延續上下文的不是聊天串，而是 repo 裡的檔案。排除了 Spec Kit（與 X.OSpec 雙重管理）、排除了 Kiro Specs（避免綁定單一 IDE）。
Five Layers

五層架構：從底座到紀律

Toolkit 自動生成 Layer 1–3（核心），Layer 4–5 可選整合。每一層解決不同的問題。

L5 Superpowers + GSD 工程紀律 + 階段式執行引擎 — 解決「品質失控」和「執行無序」
L4 X.OSpec CLI Spec 自動化工具（可選）— 自動生成 change skeleton、驗證 spec 一致性
L3 changes/ 單次變更包 — 解決「Scope 漂移」：proposal 鎖定範圍、tasks 逐一勾選
L2 specs/ Living Specs — 解決「Context 遺失」：Agent 讀 spec 就能理解系統全貌
L1 repo + git 底座 — 解決「跨工具不可攜」：純 Markdown + git，任何 IDE 都能用
核心映射：1 個 GSD Phase = 1 個 X.OSpec Change。GSD 負責「按什麼順序做」，X.OSpec 負責「做的內容是什麼」。不會有兩套追蹤系統。
Use Cases

五大操作情境

根據你的需求，選擇對應的啟動方式。

🌱

情境 1：全新專案

執行 ./setup.sh 啟動 TUI，互動填寫後生成完整骨架。

🏗️

情境 2：既有專案導入

在專案目錄執行 setup.sh --brownfield，注入框架不動原始碼。

🍴

情境 3：Fork 專案

Clone 後用 --brownfield 導入，再用 --add-change 開工。

📦

情境 4：追加 Change

在已導入的專案中，--add-change "feature-name" 新增變更包。

⚙️

情境 5：CI/CD 自動化

用 --non-interactive 加全部參數，一行指令完成。

Quick Start

三步驟快速開始

# Step 1: 取得工具 git clone https://github.com/ShiGaChenTW/xospec-toolkit # Step 2: 執行安裝 + 啟動 TUI cd xospec-toolkit && ./setup.sh # Step 3: 填完 TUI 後按 F2 生成 repo # 或使用 CLI 模式一行搞定： ./setup.sh --non-interactive \ --project-name "my-billing-service" \ --capabilities "auth,billing,notification" \ --target-dir ./my-billing-service
建議：在 ~/.zshrc 加入 alias xospec="$XOSPEC_HOME/setup.sh"，之後在任何目錄直接打 xospec 即可。
Reference

setup.sh 指令速查表

指令功能
./setup.sh完整流程：安裝依賴 + 啟動 TUI 互動介面
./setup.sh --brownfield在當前目錄導入 X.OSpec 框架（不動既有檔案）

./setup.sh --install只安裝依賴與 Skills，不啟動 TUI

./setup.sh --status顯示目前安裝狀態（Python / Skills / Hook）
./setup.sh --uninstall移除已安裝的 Skills 與 Preflight Hook

setup.sh 會自動檢查 Python 3.10+、安裝 Jinja2/PyYAML、建立 Skills symlink、部署 Preflight Hook。

Workflow

完整開發循環（8 步驟）

從確認 Phase 到完成驗收，一個 Change 的標準流程。每一步都有明確的「讀取→產出」檔案對應。

1

確認 Phase GSD

讀 ROADMAP.md 找到當前 phase，讀 STATE.md 確認無 blocker

讀 ROADMAP.md → 讀 STATE.md
2

理解需求 X.OSpec

讀 proposal.md 了解動機，讀 spec.md 理解系統現有行為

讀 proposal.md → 讀 specs/<cap>/spec.md
3

探索方案 Superpowers

觸發 brainstorming skill，列出 2–3 個方案並評估取捨

讀 proposal.md → 產出 方案比較
4

撰寫計畫 Superpowers + X.OSpec

觸發 writing-plans skill，輸出到 design.md 和 tasks.md

讀 proposal.md → 寫 design.md + tasks.md
5

TDD 實作 Superpowers

先寫測試、再寫實作；每完成一個 task 勾掉 tasks.md 的 checkbox

讀 design.md + tasks.md → 更新 tasks.md ☑️ 逐項勾選
6

程式碼審查 Superpowers

觸發 code-review skill，修復 CRITICAL 和 HIGH 問題

讀 design.md 比對實作是否符合設計
7

Spec 回寫 X.OSpec

寫 spec_delta.md 記錄行為差異，合併回 spec.md

寫 spec_delta.md → 合併 specs/<cap>/spec.md
8

完成驗收 GSD + Superpowers

觸發 verification skill，檢查 DoD，更新 ROADMAP.md 和 STATE.md

更新 ROADMAP.md + STATE.md 標記 phase 完成

檔案交互關係

proposal.md

起點檔案。被 Step 2/3/4 讀取，驅動後續所有產出

design.md + tasks.md

Step 4 產出 → Step 5 逐項執行 → Step 6 比對設計

spec_delta.md → spec.md

Step 7 寫 delta，合併回 Living Spec，完成閉環

ROADMAP.md + STATE.md

Step 1 讀取狀態 → Step 8 更新進度，GSD 閉環

Closed Loop

閉環開發循環

X.OSpec 的核心設計是一個完整閉環——每一次變更都從 Spec 出發，最終回寫到 Spec，確保規格永遠是真相來源。

#步驟說明
1讀取 SpecAgent 進入 repo 先讀 .xospec-map.md 定位上下文，再讀 specs/ 理解系統現有行為
2建立 Change在 changes/ 建立 proposal → design → tasks 三件套，明確「為什麼做、怎麼做、做什麼」
3實作 + TDDPreflight Hook 攔截：沒有 active change 就不能寫程式碼。逐一勾完 tasks.md
4寫 Spec Delta用 GIVEN/WHEN/THEN 精確記錄這次變更造成的行為差異
5Merge 回 SpecDelta 合併回 Living Spec，標記 Status: Merged，規格回到最新狀態，閉環完成
為什麼是閉環？沒有 Spec Delta 回寫，Living Spec 會逐漸過時，最終 Agent 讀到的規格跟實際程式碼不符——這正是傳統開發文件腐爛的根本原因。閉環確保「Spec 永遠等於程式碼的真實行為」。
Bundled Skills

隨附的 3 個 Claude Code Skills

setup.sh 會自動將這三個 Skill 安裝為 ~/.claude/skills/ 的 symlink，在開發過程中由 Agent 自動觸發。

xospec-generator

v2.1.0

X.OSpec 原生架構師。Agent 根據五層結構規範動態生成 spec-driven repo。

觸發時機「建立新專案」「xospec」「scaffold」「變更提案」
核心能力Greenfield 骨架生成、Change Package 建立、.xospec-map.md 維護
閉環角色負責「建立 Change」和「Spec Delta 合併」兩個環節

xospec-preflight

v1.0.0

PreToolUse Hook 守衛。攔截 Write/Edit，確保「先有 Change Package，再寫程式碼」。

觸發時機每次 Agent 執行 Write 或 Edit 時自動攔截（非手動呼叫）
檢查項目Active Change 存在？Tasks 有未完成項？全完成時警告 Scope Drift
閉環角色負責「實作階段」的紀律守衛，阻止跳過 Change 直接寫碼

xospec-brownfield-onboard

v1.0.0

既有專案導入精靈。互動式掃描現有結構、推斷 Capabilities、建立最小規格骨架。

觸發時機「導入 xospec」「brownfield」「既有專案加入 xospec」
五步流程掃描 → 推斷 Capabilities → 建立骨架 → 建立首個 Change → 提交
閉環角色負責閉環的「起點建立」——從零建立第一份 Living Spec 基準線
自動安裝：setup.sh 執行時會自動 ln -sfn 三個 Skill 到 ~/.claude/skills/，並將 preflight hook 複製到 ~/.claude/hooks/。無需手動設定。
Best Practices

日常操作建議

📄

先 Spec 再 Code

確認 spec.md 存在，沒有就先補基本規格。AI Agent 讀 spec 才知道系統「應該是什麼樣」。

📦

一個 Change 一個功能

每個 change folder 只做一件事。proposal 控制 scope，防止 Agent 自作主張加功能。

✅

Tasks 拆小拆明確

每個 task 控制在一次 Agent 執行可完成的大小。太大就再拆，別讓 Agent 猜。

🔍

Review 順序：意圖先於程式碼

PR Review 順序：proposal → design → tasks → spec delta → code diff。先確認方向對再看實作。

🔄

Spec Delta 不能忘

Change 完成後，一定要寫 spec delta 並合併回 living spec。這是維持規格一致性的關鍵。

📊

STATE.md 當日記寫

每次工作結束更新 STATE.md，記錄決策和進度。下次開工時 Agent 讀這份檔案就能接上。

Preflight Hook：安裝後，每次 Write/Edit 都會自動檢查是否有 active change package。沒有就會提醒你先建一個。
